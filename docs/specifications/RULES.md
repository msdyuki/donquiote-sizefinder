# プロジェクトルール

本プロジェクトにおけるコーディング規約、開発ワークフロー、および AI（私）に対する指示事項などのルールは、このファイルで管理します。

## 1. コーディング規約 (Coding Guidelines)
- **プラットフォーム**: コンテンツは**サイネージSTB上で動作するElectron**で再生されることを前提とする。
- HTML/CSS/JS: Electron環境で最適に動作するよう、ローカルファイル（`file://`）プロトコルでの読み込みや、オフライン環境・制限されたハードウェアリソースを考慮した軽量な実装を行う。
- フレームワーク: （※実装計画での合意に基づき、Vite等を使用する場合でも最終的にElectronにバンドルしやすい形式でビルドする）
- フォーマット: Prettierを使用し、インデントはスペース2つとする。

## 2. デザインルール (Design Rules)
（例）
- カラースキーム: ドン・キホーテのブランドカラー（黒・黄色・赤）をベースとする。
- UI/UX: リッチでダイナミックなデザイン（マイクロアニメーションなど）を優先する。

## 3. ディレクトリ構成 (Directory Structure)
（例）
- `/docs`: 仕様書やルールなどのドキュメント類
- `/src`: ソースコード（必要に応じて作成）
- `/assets`: 画像やフォントなどの静的ファイル

## 4. 開発・実行環境 (Environment)
- **OS**: 開発および実行環境（サイネージSTB）は **Windows** を前提とする。
- **シェル**: コマンドの実行やパスの指定は **PowerShell** を使用する。（例: `\` によるパス区切り、`&&` の代わりに `;` や改行を用いるなど、PowerShell特有の構文に配慮する）

## 5. AIアシスタントへの指示 (AI Instructions)
- AIとの対話や出力アーティファクトは、必ず**日本語**で行うこと。（グローバルルール適用）
- 新しい機能を実装する際は、事前に `implementation_plan.md` を作成して確認を取ること。
- コマンドを実行する際は、Windows環境およびPowerShellであることを考慮した上で提案・実行すること。
- **作業ログとGit運用**: 区切りのタイミングで作業ログを作成した際は、作業ログ自体はGit管理から除外しつつ、実装したコードなどの差分は必ずコミットおよび `git push` まで自動で行うこと。

---
※ このルールはプロジェクトの進行に合わせて自由に追加・修正してください。私がコードを書く際や設計を行う際には、これらのルールを遵守いたします。
